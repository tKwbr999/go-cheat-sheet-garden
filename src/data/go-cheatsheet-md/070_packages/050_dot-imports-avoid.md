## タイトル
title: ドットインポート (`.`) - 非推奨

## タグ
tags: ["packages", "package", "import", "ドットインポート", "非推奨", "可読性"]

## コード
```go
package main

// fmt と math をドットインポート (非推奨)
import (
	. "fmt"
	. "math"
)

func main() {
	// fmt.Println ではなく Println で呼び出せる
	// しかし、これが fmt.Println なのか、
	// main パッケージの Println なのか不明瞭になる
	Println("Hello") // fmt.Println が呼ばれる

	// math.Pi ではなく Pi で参照できる
	Printf("円周率: %f\n", Pi) // fmt.Printf も同様

	// math.Sqrt ではなく Sqrt で参照できる
	Printf("平方根: %f\n", Sqrt(2))

	// もし main にも Pi があったら衝突する
	// var Pi = 999 // エラーになる可能性
}

```

## 解説
```text
`import` 文でエイリアスとしてドット (`.`) を使う
特殊な形式を**ドットインポート**と呼びます。
**構文:** `import . "パッケージパス"`

**動作:**
インポートしたパッケージのエクスポートされた識別子を、
現在のパッケージ内で**パッケージ名を付けずに直接**
参照できるようになります (例: `fmt.Println` -> `Println`)。

**非推奨:**
コードが短く見えますが、以下の理由で**強く非推奨**です。
1.  **可読性低下:** `Println` や `Pi` がどのパッケージのものか
    判別しにくく、コード追跡が困難になる。
2.  **名前空間汚染:** インポート先の公開識別子がすべて
    現在の名前空間に入り、意図しない名前衝突のリスクが高まる
    (例: 自前の `Pi` 変数と `math.Pi` が衝突)。
3.  **保守性低下:** インポート先パッケージの更新で
    予期せぬ名前衝突が起こる可能性がある。

**例外的な利用: テストコード**
BDD フレームワーク (`ginkgo`, `gomega` 等) では、
テストコードを自然言語に近づけるため、慣習的に
ドットインポートが使われることがあります。
しかし、これも限定的なケースです。

**まとめ:**
ドットインポートは可読性・保守性を損なうため、
**通常は避けるべき**です。
`fmt.Println` のようにパッケージ名を明示することで
コードの明確さが保たれます。
